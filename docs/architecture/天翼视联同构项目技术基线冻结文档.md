> 冻结日期：2026-03-10  
> 用途：用于重新创建一个**技术体系完全相同**、但**界面形态与功能逻辑可重新设计**的新项目。

## 1. 文档定位

这不是产品方案，也不是需求说明书，而是一份**技术冻结基线文档**。  
目标是把当前项目中**不能轻易改变的技术底座**一次性钉死，便于后续：

1. 重建一个技术同构的新项目；
2. 保留与天翼视联平台、高德地图、通知链路、坐标规则、部署方式的兼容性；
3. 允许新项目在页面布局、交互路径、业务流程编排、菜单结构、视觉风格上彻底重做；
4. 避免后续重建时因为“接口签名方式、令牌策略、坐标系、流媒体协议、告警接口、持久化规则”发生偏差，导致新旧项目技术不兼容。

## 2. 一句话冻结结论

**当前项目的本质，是一个部署在用户电脑上的 .NET 8 Windows 桌面端天翼视联运维监控客户端。**  
它以**天翼视联能力开放平台**为核心外部依赖，以**高德地图**为地图承载，以**本地轻量持久化**保存少量补录与运行态数据，以**Webhook**方式向企业微信/量子密信发送通知，面向“慢直播点位运维巡检”场景，重点解决：

- 点位列表与地图态势展示
- 设备在线/离线与基础状态查询
- 视频直播/回看接入
- 普通设备告警与 AI 告警查询
- 地图坐标修正与未定位点位治理
- 运维态势统计与巡检支撑

## 4. 当前工程技术基线

## 4.1 解决方案/分层结构

从现有项目构建与命名可确认，解决方案采用典型分层方式：

- `TylinkMonitorDemo.Core`
- `TylinkMonitorDemo.Infrastructure`
- `TylinkMonitorDemo.Services`
- `TylinkMonitorDemo.UI`
- `TylinkMonitorDemo.App`

建议新项目继续保持同样的分层语义：

### Core 层
负责纯业务模型、值对象、接口定义、规则约束，不直接依赖 UI 和具体外部实现。

### Infrastructure 层
负责文件存储、本地 JSON、可能的 HTTP 外围适配、序列化、配置读写、系统目录访问等。

### Services 层
负责天翼视联 API 编排、地图数据转换、告警聚合、状态计算、通知发送、业务用例组合。

### UI 层
负责 View / ViewModel / 窗口 / 页面 / 地图页 / 告警页 / 设备页等前端展示和交互。

### App 层
负责启动、依赖注入、配置装配、全局异常、应用生命周期和入口。

## 5.2 鉴权模式冻结：用户无感知获取令牌

当前项目技术基线明确应使用**用户无感知获取令牌**，适合“后台/桌面客户端直连企业主设备资源，不要求扫码登录”的场景。

### 冻结要点

- 应用授权模式：**用户无感知获取令牌**
- `enterpriseUser` 为关键参数，代表已授权企业主手机号
- `accessToken` 有效期 7 天
- `refreshToken` 有效期 30 天
- 访问令牌不是一次一取，必须**缓存复用**
- 过期后刷新，不要每次调用接口都重新取 token

### 新项目必须遵守的令牌策略

1. 首次获取后缓存本地
2. 每次业务接口调用前检查有效期
3. 过期优先走 refresh
4. 刷新失败再重新获取
5. 严禁高频重复申请 token
6. 严禁跨应用复用 token
7. `enterpriseUser` 必须是门户里已授权的企业主手机号

## 5.3 公共请求规范冻结

所有业务 API 都不是裸请求，而是必须走平台公共参数包装。  
这部分是重建时**最不能出错**的技术底座。

### 5.3.1 通信协议

- HTTPS
- POST
- `application/x-www-form-urlencoded`
- UTF-8

### 5.3.2 公共参数

每次请求应包含：

- `appId`
- `params`
- `signature`
- `version`
- `clientType`
- `timestamp`

### 5.3.3 请求头

必须带：

- `apiVersion: 2.0`

### 5.3.4 参数加密

- `params`：使用 **XXTea** 对具体业务参数串加密
- 对称密钥：`AppSecret`

### 5.3.5 签名算法

- `signature = HMAC-SHA256(appId + clientType + params + timestamp + version, appSecret)`

### 5.3.6 版本/解密规则

- `version = "v1.0"`：通常 XXTea 解密
- `version = "1.1"`：通常 RSA 解密响应密文
- 平台资料默认推荐使用 `1.1`

### 5.3.7 不能犯的错误

- 漏传 `apiVersion: 2.0`
- 漏传公共参数
- 直接把业务参数明文发过去
- 签名串字段顺序错误
- `parentUser` 不需要时传 `"null"` 字符串
- 跨应用复用 token / authCode

## 6.3 直播流能力

项目核心运维目标之一，是判断视频是否正常、画面是否偏斜、是否能正常巡看，因此直播流能力必须保留。

### 典型协议/接口

- HLS：`getDeviceMediaUrlHls`
- RTMP：`getDeviceMediaUrlRtmp`
- WebRTC：`getDeviceMediaWebrtcUrl`
- FLV：`getDeviceMediaUrlFlv`
- H5 聚合流地址：`getH5StreamUrl`

### 冻结事实

- 浏览器/H5 场景通常更适合 HLS / WebRTC
- WebRTC 低延迟，适合实时巡看
- RTMP/H265 兼容性存在前端差异
- 部分设备默认 H265，需要考虑播放器兼容问题
- 流地址不是永久固定地址，存在时效
- 国标/社标/企标设备在流时效与参数支持上有差异

### 重建建议

新项目即便 UI 改成多宫格、态势播放墙、点位弹窗巡看，底层仍应保留：

- 协议选择能力
- 地址有效期处理
- 失败重取机制
- 静音/非静音参数控制
- 码流清晰度策略（如设备支持）

## 7.2 坐标体系冻结

这是非常关键、绝不能混乱的部分。

### 当前基线已经确认的坐标规则

1. 接口返回坐标如果是 **BD-09（百度坐标）**，在地图展示前要先转换为 **GCJ-02**
2. 本地手工导入/补录的坐标，按**地图展示坐标直接使用**
3. 地图页面最终展示，以 **GCJ-02** 作为统一展示坐标基线

### 冻结结论

- 新项目所有地图展示必须统一到 **GCJ-02**
- 不得在页面层混用 BD-09、WGS84、GCJ-02 而不做显式转换
- 后续任何导入模板、批量修正、手工补录，都必须明确提示“请按高德 GCJ-02 填写”

## 8.3 持久化位置

- 不回写天翼视联平台原始设备信息

## 8.4 坐标优先级规则

当前基线已经确认：

1. **接口坐标**
2. **本地导入/手工补录坐标**
3. **未定位**

即：

- 如果接口有有效坐标，用接口坐标
- 如果接口无有效坐标，但本地有补录坐标，用本地坐标
- 两者都无，则判定未定位

这条规则必须保留。

## 8.6 导入校验规则冻结

当前基线已经确认的校验规则：

1. `deviceCode` 不能为空
2. `manualLongitude` / `manualLatitude` 要么都空，要么同时填写
3. 坐标填写时必须是数字
4. 经度范围：`[-180, 180]`
5. 纬度范围：`[-90, 90]`
6. 不允许 `0,0`
7. 同一 `deviceCode` 多条记录时，**最后一条生效**
8. 坐标和备注都空时，视为清除本地补录记录

这些规则必须原样继承。

## 13. 已知平台限制与重建时必须考虑的问题

## 13.1 平台授权限制

- `enterpriseUser` 必须提前在门户完成授权
- 新应用如换 `AppId`，旧 token 不能继续用
- 新应用如换授权模式，调用方式也必须同步调整

## 13.2 IP 白名单与域名配置

若新项目调用或播放异常，通常要检查：

- 应用出口公网 IP 是否已加入白名单
- 业务域名/跨域域名是否已配置
- 是否误用了错误的接口路径

## 13.3 播放兼容问题

- 部分浏览器/播放器不支持 H265
- HLS 延迟高于 RTSP / WebRTC
- 流地址有过期时间，不是永久地址
- 多宫格巡看时要考虑带宽和设备解码能力

## 13.4 回调/推送问题

- 平台若回调消息，接口必须尽快返回成功
- 同步慢处理会被限流或熔断
- 解密失败通常是 RSA 私钥不匹配或版本判断错误
